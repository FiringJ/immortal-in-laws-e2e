# Plan
## Problem
- 当前“通知状态”主流程与缺陷描述/设计稿不一致：`pages/profile/index.ts` 与 `pages/invite/index.ts` 先弹自定义教程框，再由用户点击“去开启”触发 `wx.requestSubscribeMessage`，而附件 PRD 要求点击入口后直接进入“通知权限判断”，未开启时围绕系统权限弹框完成“开关 → 允许/拒绝”的交互闭环。
- `utils/notify-subscribe.ts` 在订阅模板 ID 未配置时直接返回 `{ ok: true, fallbackUsed: true }`，会在没有真实开启通知权限的情况下放行到“明日通知状态”页；这与“先判断是否已开启系统通知权限，再决定是否跳转”的需求相冲突，也是“交互错误”的高风险根因。
- `pages/notify-status/index.ts` 当前仍是占位实现：默认 `remainingDays` 为 `5`，且“增加订阅天数”按钮只弹“功能开发中”；附件 PRD 明确要求初始值为 `0`、每点击一次加 `1`、上限 `10 万天`、且不支持扣减。
- 首页底部提醒当前绑定的是“每日推荐服务/个性化推荐”而非“通知权限”：`pages/index/index.ts` 用 `settings.personalized_recommend` 控制 banner，文案也是“您未开启每日推荐服务 / 每天为你匹配对象”，与附件 PRD 中“您未开启每日推荐提醒 / 每日通知您匹配对象更新”不一致。
- 当前仓库里没有看到“今日关闭，次日 0 点继续提醒”的持久化逻辑，也没有明确的“通知权限已开启/未开启”状态缓存或查询封装；说明本缺陷不是单一弹窗文案问题，而是通知权限状态识别、首页提醒、状态页数据和失败重试链路都未完整落地。

## Impact
- 该问题位于“我的 → 通知状态”核心入口，且飞书优先级为 P0；如果用户未真正开通通知却被放行到状态页，会直接造成功能假闭环，影响后续提醒触达与用户信任。
- 若首页底部提醒继续使用“个性化推荐”状态代替“通知权限”状态，产品会在两个入口上给出互相冲突的引导：设置页像是推荐开关问题，通知状态页像是订阅消息问题，用户无法理解到底该开启哪一种能力。
- 若“增加订阅天数”仍是占位实现，即使用户成功开启通知，也无法完成 PRD 要求的核心操作，等于后续页面仍不可用。
- 失败链路若未按设计保留一级教程弹框并仅弹二级失败提醒，会导致重试体验与设计验收不一致；重复失败时也难以判断是用户点了“拒绝”、模板 ID 缺失，还是小程序 API 调用失败。
- 由于 `pages/profile/` 与 `pages/invite/` 各自复制了一套通知弹窗逻辑，本问题若只修一处，另一处会继续保留旧交互，存在双入口不一致风险。

## Reproduction Hypothesis
- 假设 1：当前线上/测试环境未配置订阅模板 ID，导致 `requestNotifySubscribe()` 走 fallback 成功分支，用户无论是否真正授权，都会直接跳到 `/pages/notify-status/index`。
- 假设 2：即使模板 ID 已配置，当前交互仍多了一个自定义“去开启”按钮步骤，和设计稿中“直接显示系统权限弹框 + 顶部教程蒙层”的交互不一致，因此被提报为“通知权限交互错误”。
- 假设 3：`requestSubscribeMessage` 返回 `reject` 时，现有代码统一视为失败，但没有区分“用户拒绝”“模板配置问题”“API fail 回调”三类场景；这会让二级失败弹窗无法对齐 PRD 的预期触发条件。
- 假设 4：通知状态页的默认天数和按钮行为尚未实现，导致即便前置授权成功，实际落地页面也与图 2 PRD 不一致，从而放大了“通知权限流程错误”的整体感知。
- 假设 5：首页底部提醒功能尚未从“每日推荐服务”迁移为“通知权限提醒”，因此 PRD 中的 b 分支（首页固定提醒、点击叉号当日不再提醒、次日恢复）实际上仍未开发。

## Evidence To Collect
- 在微信开发者工具或真机复现“我的/邀请页 → 通知状态”入口，记录 `wx.requestSubscribeMessage` 实际返回值，区分 `accept`、`reject`、`ban`、`fail` 等结果，并截图当前 UI 与附件 `agent-memory/defect-reports/attachments/bug_4224234a1639/image.png` 对比。
- 检查运行时是否配置了订阅模板 ID：包括 `app.globalData.notifySubscribeTemplateIds` 和本地存储 `notify_subscribe_template_ids`；若两处都为空，可直接验证 fallback 误放行是否存在。
- 对照设计稿 `design/3.设计稿/神仙亲家-我的-通知状态-弹窗1.png`、`design/3.设计稿/神仙亲家-我的-通知状态-弹窗2.png`、`design/3.设计稿/神仙亲家-我的-通知状态-开启失败-弹窗1.png`、`design/3.设计稿/神仙亲家-我的-明日通知状态1.png`，核对四个关键态：未开开关、开关已开待允许、拒绝后失败弹窗、授权成功后的状态页。
- 结合仓库待办文档 `docs/0. 代办/2026.2.1-2.md`，确认当前实现是否只是早期占位版本；如果是，需要把 bug 定性为“设计流程已立项但实现未闭环”，不是单纯样式偏差。
- 补查首页提醒现状：确认 `pages/index/index.ts` 的 banner 是否仅由 `personalized_recommend` 控制，且关闭后是否只在当前会话隐藏；这将决定修复时是否需要新增“按天关闭提醒”的本地持久化键。
- 如需避免误判平台能力，应在真机验证“教程蒙层 + 系统订阅弹框”能否同时满足交互稿；若微信系统弹框层级限制导致无法 1:1 还原，需要把平台限制单独记录给产品确认。

## Initial Fix Direction
- 第一优先级先修“假成功”根因：移除或收紧 `utils/notify-subscribe.ts` 的 fallback 放行逻辑，在模板 ID 缺失或 API 不可用时不要直接跳状态页，而是显式进入失败/不可用分支并打日志。
- 将“通知状态”流程拆为统一能力层，而不是分别写在 `pages/profile/` 和 `pages/invite/`：建议抽一个共享的通知权限控制器（状态查询、拉起订阅、失败原因归类、成功跳转），两处入口只负责触发，避免再次分叉。
- 按 PRD 重做“明日通知状态”页最小可用闭环：初始天数 `0`、点击“增加订阅天数”本地即时加 `1`、上限 `100000`；若后端后续接管，可再把本地状态替换为接口数据。
- 首页底部提醒需要从“个性化推荐 banner”中解耦，改成基于“通知权限未开启”展示，并补上“去开启”进入通知权限流程、“叉号仅隐藏到次日 0 点”的本地策略。
- 失败链路要显式覆盖三种情形：用户拒绝、系统 API 调用失败、模板配置缺失；UI 上至少要保证失败弹窗与一级教程弹框的关系和设计稿一致，便于重试。
- 修复前先补一轮可验证用例：未配置模板 ID、已配置模板但用户拒绝、用户允许成功、首页 banner 点击“去开启”、首页 banner 点击“叉”后当日不再提醒、次日重新出现；这样才能避免只修入口不修后续页面。
