## Cleaned Problem
`pages/history/index` 的“日期筛选”当前仍是页面内展开的原生 `picker mode="date"` 按钮式实现。该补充缺陷要求沿用现有弹框 UI 体系，把历史推荐的日期筛选改成截图中的底部滚筒弹层：三列年/月/日滚轮、居中高亮选中行、圆角白色面板、底部“取消筛选 / 确定推荐日期”双按钮，整体排版和交互需与截图一致。结合同源父缺陷，可将问题归一为：历史推荐只保留“单日日期筛选”，默认选中前 31 天，用户在滚轮中调整日期后，通过显式确认再应用到推荐列表。

## Product Context
- 页面拓扑显示，用户从 `pages/index/index` 顶部入口进入 `pages/history/index`；历史推荐是首页高频二级页，日期筛选是该页首屏主控件之一。
- `project-knowledge.md` 已把历史推荐列为有专门验证路径的页面，并明确校验首屏标题及 `昨日推荐 / 日期筛选` 入口，说明该筛选控件是页面体验的核心组成部分。
- 现有历史推荐页还承担会员可见性、空状态、分页加载等行为，日期筛选是切换推荐记录集合的主入口；如果控件样式和交互不对，用户会直接感知为页面主能力未完成。
- 该问题属于高优先级 UI/交互缺陷，但不是“待产品补定义”的空泛意见，已经足够进入后续计划阶段。

## Technical Context
- `pages/history/index.wxml` 当前在 `showDateFilter` 为真时渲染一段 `.date-filter`，其核心控件仍是原生 `<picker mode="date">`，不是自定义滚筒弹层。
- `pages/history/index.ts` 的 `onToggleDateFilter()` 只是展开/收起页内筛选区；`onDateChange()` 在原生 picker 返回值后立即写入 `selectedDate`、重置列表和页码，并直接 `loadData()`，这与截图中的“先滚动选择，再点击确认”交互并不一致。
- 同一文件里，`onLoad()` 已把默认日期初始化为“今天前 31 天”，`loadData()` 也已经用 `startDate = endDate = selectedDate` 做单日查询，因此父问题里“默认前 31 天、仅单日筛选”的业务规则已有部分基础。
- `pages/history/index.wxss` 目前只实现了 `.date-picker-btn` 这类轻量按钮样式，没有底部遮罩、弹层容器、滚轮选中态、双 CTA 等所需样式面。
- 仓库内已有可参考的自定义滚轮弹层：`pages/filter/index` 现使用 `picker-view` + 底部弹层来实现年份/身高范围选择；`project-knowledge.md` 也明确提示原生 `picker` 难以满足视觉还原，优先用自定义 `picker-view` 弹层。
