## Cleaned Problem
当前条目不是一条可直接落计划的单点缺陷，而是一条“记录页 6 个 tab 全量按 PRD 对账”的总核对项：`我看过 / 看过我 / 我收藏 / 收藏我 / 我解锁 / 解锁我` 六个列表页的会员阻断、跳转和弹框逻辑需要逐一比对 PRD。

就原始来源本身而言，它没有说明：
- 当前到底是哪一个 tab、哪一个入口、哪一种会员身份下出现了错误；
- 当前错误表现是“错误跳页 / 错误弹框 / 错误会员等级 / 错误底部常驻条 / 错误按钮状态”中的哪一种；
- 是否希望作为一个汇总父单跟踪，还是要继续拆成可实现的子缺陷。

因此，这更像一条“会员阻断专项清单/总问题”，而不是一个边界清晰的单一 bug。

## Source Quality

## Product Context
- 该条目对应 `pages/message-record/index`，消息页金刚区会进入 3 组记录页：浏览记录（`看过我 / 我看过`）、收藏记录（`收藏我 / 我收藏`）、解锁记录（`解锁我 / 我解锁`）。
- PRD `design/1.需求文档/神仙亲家--小程序产品需求文档.md:141` 到 `design/1.需求文档/神仙亲家--小程序产品需求文档.md:144` 给出了 6 个 tab 的阻断矩阵，核心差异包括：
- `看过我`：非会员点击卡片、详情按钮、联系按钮都应被会员阻断，底部常驻“开通会员 查看全部资料”。
- `我看过`：非会员应看到高斯模糊卡片，点击模糊区域跳会员开通页；会员才可正常进详情/私信。
- `收藏我`：非会员也应统一会员阻断，且底部提示文案应为“共有 xxx 人收藏我”。
- `我收藏`：无论是否会员，卡片和详情按钮都可进详情；“联系对方/已解锁”进入私信页，并在私信页底部执行会员阻断。
- `解锁我`：旧 PRD 写的是“会员 / 24 小时内被对方解锁 / 任一方已发过私信”满足其一才放行，否则阻断。
- `我解锁`：旧 PRD 写的是无论是否会员，都可正常进详情/私信，不受会员阻断。
- 但 tracker 中已存在一条更晚的明确需求变更：`wf-2026-03-13T18-18-12-216Z-r58x6y` 要求 `解锁我` tab 取消会员阻断，并把“24 小时内免费查看”降为纯提示文案。这与当前“以 PRD 为准”的说法直接冲突。

## Technical Context
- 当前前端会员阻断逻辑集中在 `pages/message-record/index.ts`，但实现方式是按分支硬编码，而不是显式维护 6-tab 规则矩阵：
- `showVipBanner` 只覆盖 `recordType=view + direction=me_to_other`，且使用 `memberStore.canViewHistory()`，实际把 `我看过` 做成了偏“至尊会员”限制，不是单纯“非会员限制”。
- `showBottomBar` 只覆盖 `view/favorite + other_to_me`，说明当前底部阻断条只对 `看过我 / 收藏我` 两个 tab 做了特殊处理。
- `onRecordTap()` 只对 `看过我`、`收藏我` 的非会员详情点击做拦截；`解锁我 / 我解锁` 不在该分支内。
- `onContactTap()` 中，`解锁我 / 我解锁` 两个 tab 当前都被直接放行到 `pages/chat/index`；`我收藏` 则有专门的“剩余免费解锁次数 / 会员 / 已建立联系”分流；其余非会员回落到 `member-unlock-modal`。
- `store/memberStore.ts` 已提供 `canViewRecords()`，但记录页没有以此为统一开关，说明当前页面权限矩阵分散在页面事件里，后续若要“按 PRD 全量对齐”需要先统一规则来源。
