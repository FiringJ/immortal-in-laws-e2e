# Plan
## Problem
- 当前仓库虽然已经接入了邀请入口，但 `pages/invite/index` 实际实现的是“邀请家人帮孩子找对象”页面，不是 PRD 中的“分享领现金 / 邀请有奖”功能页。页面主体由资料卡、会员 banner、家庭成员卡、常用功能组成，标题文案也是“邀请家人帮孩子找对象”，与“累计邀请数 / 获得红豆数 / 获得现金数 / 收益明细 / 我的邀请 / 现金提现”完全不一致。证据见 `pages/invite/index.wxml:17`、`pages/invite/index.wxml:61`、`pages/invite/index.ts:2`。
- 当前页面的数据加载只调用了 `generateInvite()` 和 `fetchFamilyAccounts()`，没有邀请统计、收益明细、我的邀请列表、催认证、提现、提现记录等任何“邀请有奖”所需接口。证据见 `pages/invite/index.ts:302`、`pages/invite/index.ts:313`、`services/family.ts:73`。
- 当前交互也停留在“分享家庭邀请码”层面：点击 invite 功能仅 `showShareMenu + toast`，分享时只返回 `/pages/invite/index?code=...` 的小程序路径；没有“活动规则”“去提现”“收益明细分页”“我的邀请列表”“催 ta 认证”等逻辑。证据见 `pages/invite/index.ts:434`、`pages/invite/index.ts:463`。
- PRD、待办拆解和设计稿都明确要求“邀请有奖”是一个独立完整模块，但前端实现未覆盖这些要求。证据见 `design/1.需求文档/神仙亲家--小程序产品需求文档.md:210`、`docs/prd-todo.md:218`、`design/3.设计稿/神仙亲家-我的-分享领现金-收益明细1.png`、`design/3.设计稿/神仙亲家-我的-分享领现金-我的邀请1.png`、`design/3.设计稿/神仙亲家-我的-分享领现金-现金提现1.png`。
- API 文档当前只明确暴露了 `/api/v1/invite/code` 这一类“生成好友邀请码”能力，未见“邀请统计 / 收益明细 / 我的邀请 / 催认证 / 提现 / 提现记录”接口定义，说明问题很可能是前后端链路整体未落地，而不是单纯页面漏接。证据见 `docs/默认模块.openapi.json:2384`、`design/2.接口文档/接口文档.md:3315`。
- 文档状态与真实实现存在偏差：`README.md` 已将“邀请好友”标记为完成，容易误导排期和验收。证据见 `README.md:158`。

## Impact
- 这是直接影响拉新与变现的核心模块缺失：用户从“我的”页进入后无法看到邀请奖励数据，也无法完成提现或查看收益，和飞书 P0 优先级一致。
- 红豆任务页已经把 `invite_register` / `invite_member_buy` 等任务导流到 `pages/invite/index`，但落地页并未提供对应奖励闭环，会造成任务入口存在、功能结果缺失的断层。证据见 `pages/beans/index.ts:351`。
- 由于当前邀请页展示的是“家庭协作找对象”，用户会把“邀请有奖”和“邀请家人”两个业务混淆，影响文案理解、设计验收和后续联调。
- 若后端确实未提供完整接口，则前端即使补 UI 也无法完成真实闭环；需要尽早把问题定位为“功能未实现”而不是“单点页面缺陷”。

## Reproduction Hypothesis
- 假设 1：从“我的”页点击“邀请好友”后，进入的是错误页面模板。当前入口已存在，但目标页沿用了“邀请家人帮孩子找对象”的实现，而不是“分享领现金”页。证据见 `pages/profile/index.ts:410`、`pages/profile/index.ts:411`。
- 假设 2：当前需求拆分中 `INV-01 ~ INV-04` 尚未实施，仓库只完成了最早期的邀请码分享壳子，因此飞书反馈“目前该功能未实现”是准确描述，而非局部显示问题。证据见 `docs/prd-todo.md:220`、`docs/prd-todo.md:223`、`docs/prd-todo.md:226`、`docs/prd-todo.md:229`。
- 假设 3：后端当前只支持邀请码生成，不支持邀请奖励统计和提现相关能力，所以前端实现才停留在 `services/family.ts` 的 `generateInvite()`。这意味着修复需要同时补接口契约和前端页面，不是前端单文件微调可解。
- 假设 4：分享回流链路也不完整。虽然 `onLoad(query)` 会读取 `code` 参数，但 PRD要求“未注册/注册未完成/已注册”进入不同落点，当前页面没有看到这类路由分流与鉴权兜底逻辑，说明被邀请者落地流程也未完成。证据见 `pages/invite/index.ts:282`、`design/1.需求文档/神仙亲家--小程序产品需求文档.md:214`。

## Evidence To Collect
- 在微信开发者工具复现“我的 → 邀请好友”，截图当前页首屏，确认实际展示的是“邀请家人帮孩子找对象”而不是“分享领现金”。
- 抓取进入邀请页时的网络请求，确认当前只会请求邀请码和家庭账号数据，记录是否出现 `/api/v1/invite/code` 之外的邀请奖励相关请求。
- 对照设计稿 `design/3.设计稿/神仙亲家-我的-分享领现金-空1.png`、`design/3.设计稿/神仙亲家-我的-分享领现金-收益明细1.png`、`design/3.设计稿/神仙亲家-我的-分享领现金-我的邀请1.png`、`design/3.设计稿/神仙亲家-我的-分享领现金-现金提现1.png`，整理缺失的页面态、弹窗态和列表态。
- 和后端对齐接口现状：至少确认是否已有或计划提供以下能力——邀请统计、收益明细分页、我的邀请分页、催认证、提现、提现记录、活动规则；若没有，需要把缺陷升级为“需求模块未开发完成”。
- 和产品确认奖励口径：PRD 中 A/B 红豆数待定，`docs/prd-todo.md` 又写了“邀请好友注册奖励 100 豆/人（可配置）”，需要在开发前统一奖励数值与文案。
- 核对“邀请家人”和“邀请有奖”是否必须拆成两个入口；当前 `pages/invite/` 同时承载两类语义，存在信息架构冲突，需要产品给出最终导航方案。

## Initial Fix Direction
- 第一优先级先做范围确认：把该问题定性为“邀请有奖模块整体未落地”，而不是单纯 UI 偏差；在缺陷结论中明确当前仅实现了“邀请码分享 / 家庭邀请”子能力。
- 若按现有 PRD 修复，建议拆成 4 个实现子任务，对齐 `INV-01 ~ INV-04`：
  - `INV-01`：重做 `pages/invite/index` 为“分享领现金”首页，接入邀请统计、分享标题、未登录/未注册回流处理。
  - `INV-02`：新增收益明细列表与分页，按时间倒序展示红豆/现金收益，遵守分页从 `0` 开始的仓库规则。
  - `INV-03`：新增“我的邀请”列表、实名认证进度、催认证按钮与提醒状态，并打通消息通知接口。
  - `INV-04`：新增现金提现页/记录页或同页分区，接入可提现金额、固定档位提现、提现记录查询。
- 服务层建议不要继续把邀请能力塞在 `services/family.ts`；应新增独立 `services/invite.ts` 与对应 `types/` 定义，避免“家庭邀请”和“邀请有奖”继续耦合。
- 接口未齐前可先做前端结构重构，但不要再用当前家庭邀请页面冒充邀请有奖；若后端暂缺接口，至少在页面显式展示“功能建设中”，并关闭红豆任务对该页的误导入口，避免继续向用户暴露假闭环。
- 同步修正文档：把 `README.md` 中“邀请好友已完成”改为“邀请有奖待完成 / 家庭邀请已接入”，并在任务排期中标注依赖 `APIWRAP-09`，防止重复误判状态。证据见 `docs/plan.md:81`。
